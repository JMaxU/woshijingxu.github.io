redis3.0集群

发表于 2016-03-22   |   分类于 技术   |  

一些问题

  1. redis3.0集群是如何做数据分片的? 相比之前常用的方式有什么优点?
  2. redis3.0集群是如何做高可用的? 相比之前的哨兵有什么优势?
  3. redis3.0集群的性能如何? 相比单点是不是有所降低?

redis集群分片

分片的原因:

  1. 由于redis是单线程的, 只能用到一核. 在单机上启动多个redis实例, 充分利用多核机器.
  2. 访问量增大到一定程度后, 单机io网卡, cpu抗不住.
  3. 存储数据量增加到一定程度后, 单机内存不够, 需要存储到多个机器上.

redis3.0之前方式

client

client进行对key进行hash计算, 分到某一实例.

这种方式有以下缺点:

  1. client端对hash的耦合. 由于不同的client的hash算法可能有区别, 所以想切换别的client会有点困难.
  2. 如果redis的作用是当成数据库而不是cache, 那当增加节点与减少节点时, 不管用哪个hash算法, 势必会让已有的部分存储无法被hash访问到.

proxy

client请求到代理, 代理进行hash计算, 再分配到某一实例.
这种方式有以下缺点:

  1. 性能有降低.
  2. 同上述client方式第二点.

redis集群方式

redis3.0集群没有用之前常用一致性hash, 而是把整个集群分成了16384个hash槽.
key会通过crc16(key) modulo 16134来决定存在哪个hash槽上.
这些hash槽分布到redis各节点上.

比如有A,B,C三个节点.

  • A有0-5500号hash槽
  • B有5501-11000号hash槽
  • C有11001-16383号hash槽

新增一个master节点加入集群中时, 此节点是没有hash槽的, 需要手动添加.
同理, 删除一个master节点时, 首先要保证此节点上没有hash槽, 需要手动把此节点上的hash槽移到其他节点上.
如果一个master节点挂掉了, 而此master节点没有任何slave, 那么集群会停止工作.

当client连接集群时, 会把集群信息(哪个节点有哪些槽)缓存起来.
当集群中hash槽位发生改变, client会收到moved命令, 这时会刷新缓存.

当移到hash槽的过程中, redis cluster仍然能正常提供服务.即使key属于所移到的hash槽.

这个方式优点:

  1. 性能好.由于没有跳转代理, 由于client会缓存节点信息, 所以访问集群中单个节点相当于访问standalone redis一样快速.
  2. 支持redis cluster的client hash都将一致. redis cluster制定了统一的协议. 可以方便切换client.
  3. redis作为存储, 可以方便的增加节点和删除节点.

redis集群高可用

redis cluster中支持每个master节点有slave节点. 这个基本与哨兵雷同.

但redis高可用多了个特性. 叫做从库迁移
主要思想是如果某个masterA拥有多个slave节点, 而集群中有一个masterB没有任何slave节点(可能是挂光了),
那么slave会从多的一方移到少的一方.这使得集群更加高可用.
